Reproducibility contract (effective pass-4 re-record)
=====================================================

From the pass-4 re-record forward, the recorded
`content_integrity.wasm` + `content.wasm` + `humm_earth_core.dna` +
`humm-earth-core-happ.happ` sha256s are produced by a reproducible
build pipeline. Building the SAME commit from ANY filesystem path
on ANY host with the pinned toolchain MUST produce byte-identical
wasms (and therefore byte-identical DNA + .happ hashes).

Why two drift sources existed and what the pipeline removes:

- The wasm `name` custom section embedded LLVM codegen-unit
  suffixes (`.llvm.NNNNNNNNNNNN`) that hash the ABSOLUTE PATH of
  the crate being compiled. Two contributors building the same
  commit from different paths got two different `name` sections,
  two different wasm shas, two different DNA hashes. Binaryen's
  `wasm-opt --strip-debug` removes this section (binaryen treats
  it as part of debug info).
- The wasm `producers` custom section embedded the rustc version
  string (compiler hash + date). Stable within a toolchain version
  but drifts across upgrades. `wasm-opt --strip-producers` removes
  it.

Both sections are no-op for execution: the wasm spec marks custom
sections as having no semantic effect, and wasmer (Holochain's
runtime) ignores them.

`--remap-path-prefix` flags in the build script are defense-in-
depth — this project's `panic = abort` (the wasm32-unknown-unknown
default) + `opt-level = "z"` combination DCEs `file!()`/registry
path strings before they hit the wasm, so the remap is largely a
no-op today; the flags stay correct if anyone later flips those
settings. `codegen-units = 1` is also defense-in-depth (and is
required for the strip + remap path to converge deterministically
when the codegen-unit-count would otherwise have introduced
further per-host variance).
